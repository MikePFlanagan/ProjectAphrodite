# Project Aphrodite — API Routes, Chat Engine, and Relationship Engine Review

**Review status:** Complete  
**Area score:** 7/10  
**Ship status for this area:** Functional MVP foundation; harden reliability, safety, and concurrency before broad public use.

## Executive Summary

The chat pipeline is coherent and works end to end:

- Requests are authenticated.
- Input is validated with Zod.
- Conversation ownership is enforced.
- Daily plan limits exist.
- User and assistant messages persist.
- Mock and OpenAI providers are supported.
- Usage is recorded.
- Retention summaries and memories are generated periodically.
- Relationship scores persist and affect the visible relationship label.

The largest risks are operational:

- Quota enforcement is race-prone.
- Messages are persisted before provider configuration is validated.
- Streaming completion persistence needs stronger recovery semantics.
- Several API routes can return 500 on malformed JSON.
- There is no API-wide rate limiting or request correlation.
- The relationship evaluator is intentionally shallow and can be gamed.
- Safety relies mostly on prompting rather than a layered policy system.

## API Route Findings

### Strength — Authentication and ownership are usually explicit

Conversation, memory, creator, favorites, and billing routes generally authenticate first and scope sensitive queries to `session.user.id`.

### High Priority — Standardize malformed JSON handling

Some routes use:

```ts
await request.json().catch(() => null)
```

Others call `await request.json()` directly. The latter can throw and produce an internal error for invalid JSON.

**Recommended change:** Introduce a small route utility for safe JSON parsing and uniform validation responses.

### Medium Priority — Standardize error contracts

Routes return different shapes:

- `{ error }`
- `{ success: false, error }`
- `{ ok: true }`
- `{ received: true }`

**Recommended change:** Define a minimal API response convention with:

- Stable error code
- Human-readable message
- Optional field details
- Request/correlation ID

### High Priority — Add API rate limits

Authenticated access alone does not prevent abuse. Rate-limit at least:

- Chat
- Image generation
- Creator reference upload
- Login/signup
- Stripe checkout creation

### Medium Priority — Add observability

Console errors are insufficient for production diagnosis.

**Recommended change:** Add structured logging with:

- Request ID
- User ID or hashed identifier
- Route
- Provider
- Duration
- Error code
- Never raw credentials, private memory text, or full prompts by default

## Chat Engine Findings

### Strength — Ownership and input bounds are correct

The chat route verifies that the conversation belongs to the active user and limits user content to 8,000 characters.

### High Priority — Make quota enforcement atomic

Counting messages and then inserting later is vulnerable to concurrent requests.

**Recommended change:** Use a daily usage ledger or counter with an atomic increment and a uniqueness key based on user and UTC day.

### High Priority — Validate provider before persisting the user message

The route persists the user message and relationship update before checking whether `AI_PROVIDER` is supported.

This causes a saved message with no answer when configuration is invalid.

**Recommended change:** Resolve and validate the provider before the persistence transaction, while retaining the deliberate behavior that temporary provider failures may leave the user message saved.

### Medium Priority — Clarify streaming persistence guarantees

The assistant response is saved in `onFinish`. If a serverless function is terminated, the client disconnects unexpectedly, or a provider stream fails late, the database may contain a user message but no assistant response.

**Recommended change:**

- Create a generation-attempt record.
- Track statuses such as `PENDING`, `STREAMING`, `COMPLETED`, and `FAILED`.
- Persist partial/final output deliberately.
- Provide retry semantics that do not duplicate user messages.
- Add reconciliation for abandoned attempts.

### Medium Priority — Prevent duplicate submissions

The client disables its form while one request is streaming, but retries, multiple tabs, or network replays can create duplicate messages.

**Recommended change:** Accept an idempotency key generated by the client and enforce uniqueness per user/conversation.

### Medium Priority — Improve retention scheduling

Retention refresh runs every fifth user message inside the chat completion callback. This increases completion work and couples user-facing response handling to memory extraction.

**Recommended change:** Move retention to a background job or durable queue when production infrastructure exists. For the current MVP, record a pending retention task and process it outside the stream lifecycle.

### High Priority — Add layered safety controls

The system prompt includes valuable rules, but prompting alone is not a complete safety boundary.

**Recommended change:**

- Add input/output policy classification.
- Add age and sexual-content policy enforcement appropriate to the product.
- Add crisis and self-harm handling.
- Add report/block tools.
- Store only safety metadata needed for enforcement.
- Define model-provider safety configuration.

### Medium Priority — Avoid sending excessive retained content

The prompt uses up to 40 prior messages, summary data, and up to 12 memories.

**Recommended change:** Add explicit token budgeting and truncate by tokens rather than message count alone.

## Relationship Engine Findings

### Strength — The interface is clean

The evaluator, score updater, and label function are independently testable and separated from the web route.

### High Priority — Fix lost updates

The route reads current scores, computes absolute next scores, and writes them. Concurrent messages can overwrite each other's updates.

**Recommended change:** Apply deltas atomically or serialize updates in a transaction.

### Medium Priority — Replace hard-coded regexes with data-driven rules

The evaluator recognizes only a few words. It may misclassify quoted insults, jokes, negations, sarcasm, or messages about third parties.

**Recommended change:** Preserve the current engine interface while moving patterns and score deltas into a data structure with:

- Rule ID
- Pattern set
- Context exclusions
- Score delta
- Cooldown
- Maximum contribution per message
- Tests

This aligns with the planned data-driven evaluator and allows a later AI evaluator behind the same interface.

### Medium Priority — Comfort rises on every non-hostile message

Neutral messages increase comfort by one. Long conversations therefore trend upward even without meaningful relationship behavior.

**Recommended change:** Make passive drift intentional, capped, and slower, or remove it.

### Medium Priority — Relationship labels use an unweighted average

All six dimensions contribute equally. A character could be labeled close even with poor trust if other dimensions are high.

**Recommended change:** Define label criteria based on required floors and weighted dimensions, not only an average.

### Medium Priority — No decay, recovery, or event history

Only current scores are stored.

**Recommended change:** Add a relationship-event ledger before sophisticated behavior is introduced. This supports debugging, explainability, tuning, and rollback.

## Suggested Sprint

### Sprint title

**Chat Reliability and Relationship Engine Beta**

### In scope

- Shared API parsing and error utilities
- Atomic daily quota enforcement
- Chat idempotency
- Generation-attempt state
- Provider validation before persistence
- Structured logging
- Token-budgeted context assembly
- Relationship atomic updates
- Data-driven relationship rules
- Expanded unit and integration tests
- Initial layered safety policy boundary

### Out of scope

- Replacing the relationship engine with an LLM
- Multi-provider orchestration platform
- Full distributed queue system
- Voice chat
- New chat features unrelated to reliability

## Acceptance Criteria

- Malformed JSON never produces an unhandled 500.
- Concurrent requests cannot bypass daily limits.
- Duplicate idempotency keys cannot create duplicate user messages.
- Every chat attempt has an observable final state.
- Unsupported provider configuration is detected before message persistence.
- Relationship deltas are concurrency-safe.
- Relationship rules are data-driven and thoroughly tested.
- Context assembly has an explicit token budget.
- Safety policy tests cover prohibited and high-risk scenarios.
- Chat integration tests cover provider success, provider failure, disconnect, retry, limit exhaustion, and retention scheduling.

## Definition of Done

The chat system is ready for the next stage when it behaves predictably under retries, concurrent requests, provider failures, and malformed input; relationship updates cannot be lost; and safety enforcement is more than a system-prompt instruction.
